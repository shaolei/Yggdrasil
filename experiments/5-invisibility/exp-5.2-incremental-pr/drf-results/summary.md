# Summary: Experiment 5.2 — Incremental PR Maintenance (DRF)

## Aggregate Metrics

| Metric | Value | INVEST Threshold | Result |
|--------|-------|-----------------|--------|
| Precision | **100%** | >= 90% | PASS |
| Recall | **81%** | >= 70% | PASS |
| Decision capture | **60%** | >= 50% | PASS |
| False positive rate | **0%** | < 10% | PASS |

**All four thresholds met. Verdict: INVEST.**

## Key Findings

### 1. Precision is excellent (100%)

The conservative auto-patching strategy produced zero false positives. Every proposed update was correct and matched the expert patch. This means an automated CI integration would not introduce graph errors.

### 2. Recall is good but not perfect (81%)

The auto-patcher missed 4 of 21 required updates (19%). All misses followed two predictable patterns:
- **Secondary artifact miss** (3/4): When an internals change also affects the interface contract, the auto-patcher updates one but misses the other.
- **Responsibility boundary miss** (1/4): When a change expands what a component is FOR, not just how it works, the auto-patcher misses the responsibility.md update.

### 3. Decision capture is adequate (60%)

3 of 5 decisions were captured. The 2 misses were decisions not explicitly stated in PR text — they required inferring rationale from code patterns. This is the expected boundary: explicit decisions in commit messages are capturable; implicit decisions from code reasoning are not.

### 4. Commit type reliability

| Type | Recall | Notes |
|------|--------|-------|
| Aspect-affecting | 100% | Best — clear signal, all artifacts updated |
| Bugfix (clear) | 100% | When fix has obvious contract implications |
| Feature (simple) | 100% | Single-artifact additions |
| Bugfix (subtle) | 50% | When fix changes failure modes (interface miss) |
| Refactor | 75% | Behavioral changes disguised as refactors |
| Interface change | 67% | Misses responsibility boundary shifts |

### 5. Error patterns are systematic and addressable

Both FN patterns can be mitigated with additional rules:
1. "Check interface.md whenever internals.md is updated for new failure modes"
2. "Check responsibility.md when a diff adds new behavior categories"

These rules could raise recall to ~95% without sacrificing precision.

## Recommendations

1. **Build the CI integration.** 100% precision and 0% FP rate means automated patches will never degrade the graph.
2. **Add secondary-artifact checking rules.** The two identified patterns are systematic and rule-based.
3. **Flag implicit decisions for human review.** Do not try to infer rationale — instead, surface "this change may contain an undocumented decision" for the author.
4. **Use PR templates.** Prompting authors to explain "why this approach" would increase decision capture from PR text.

## Limitations

- Single repo (DRF). Need to validate against 2 more repos per methodology.
- Simulated PR analysis (git commits, not full GitHub PR with review comments). Real PRs with discussion threads would likely improve decision capture.
- Expert and auto patches generated by the same agent in the same session. True separation of roles would require independent sessions.
