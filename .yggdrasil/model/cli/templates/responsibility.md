# Templates Responsibility

Default templates and platform rules — used by `yg init`.

**In scope:**

- **default-config.ts:** `DEFAULT_CONFIG` — YAML string for default config.yaml. Node types: module, service, library, infrastructure. Three artifacts: responsibility.md (always required, structural_context), interface.md (required when has_incoming_relations, structural_context), internals.md (never required). Quality thresholds for min_artifact_length, max_direct_relations, context_budget.
- **platform.ts:** `installRulesForPlatform(projectRoot, platform)` — writes rules to platform-specific path; `PLATFORMS`, `Platform` type. Unknown platform falls through to generic.
- **rules.ts:** `AGENT_RULES_CONTENT` — canonical agent rules (operating manual); hand-tuned, do not generate programmatically. Used by platform.ts. Split internally into three cognitive sections (`CORE_PROTOCOL`, `OPERATIONS`, `KNOWLEDGE_BASE`) optimized for LLM attention patterns, concatenated into a single export. Core Protocol covers quick start loop, five core rules (including "capture why NOT" with rejected alternatives), failure states, escape hatch, and environment check. Operations covers conversation lifecycle (using `yg preflight`), modify code/graph checklists, greenfield graph-first workflow (6-step: aspects → flows → nodes → build-context → implement), reverse engineering with aspect identification guidance (3-instance heuristic, domain/architectural/concurrency taxonomy, step 6b for code anchors in node.yaml), bootstrap mode, drift resolution (with drift triage prioritization), graph audit (consistency + completeness checks including `aspect_exceptions` verification), and error recovery. Includes "BEFORE ENDING ANY RESPONSE" self-audit: did I modify code? If yes → did I update graph in this same response? Knowledge Base covers graph structure (with node type guidance including `infrastructure` type for guards/middleware/interceptors), artifact structure (3-level: responsibility/interface/internals), context assembly, information routing (with decisions.md rejected-alternatives template), creating aspects/flows (with `aspect_exceptions` guidance for node-specific deviations), operational rules with enhanced completeness test, CLI reference, and quick routing table.
- **graph-schemas/:** node.yaml, aspect.yaml, flow.yaml — schemas for each graph layer, copied to .yggdrasil/schemas/ during init

**Out of scope:**

- Init logic (cli/commands/init)
- Templates in .yggdrasil/schemas/ (generated by init, not part of source)
