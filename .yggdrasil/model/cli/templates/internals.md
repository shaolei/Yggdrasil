## Decisions

# Templates Decisions

**Hand-tuned content:** Agent rules content is explicitly maintained by humans. The rules.ts comment states: "Canonical agent rules content — hand-tuned, do not generate programmatically." This ensures the operating manual reflects deliberate design decisions.

**Motivation-first opening:** The EXTREMELY-IMPORTANT block leads with WHY the graph matters — "the graph exists so the user does not have to explain the same thing twice" — rather than authority ("YOU DO NOT HAVE A CHOICE"). This reframe targets the root cause of agent non-compliance: agents under task pressure drop procedural rules but retain motivational framing that connects to their core drive (being helpful). The opening also introduces `yg build-context --file` as the single pre-action command, reducing the decision tree to one step.

**Intent capture mandate:** The rules enforce "CAPTURE INTENT, BUT NEVER INVENT IT" — agents must permanently record the user's "why" (business reason, rationale) in graph artifacts when provided, but never infer or hallucinate it. This makes the graph the semantic meaning layer (Intent -> Graph -> Outputs) and protects against chat context evaporation.

**Enrichment priority guidance:** Rules now include explicit guidance on artifact enrichment order: interface.md first (highest cross-module ROI), then responsibility.md, then internals.md. This is based on experiment findings about cross-module value.

**Aspect stability tiers:** Rules include guidance on the `stability` field in yg-aspect.yaml (schema/protocol/implementation) with calibrated review urgency for each tier, and guidance on using code anchors (declared per-node in `yg-node.yaml` `anchors` field, validated by `yg validate` W014/E019) for staleness detection during drift resolution.

**Action recognition rule:** Rules include an explicit "Recognizing Graph-Required Actions" section that tells agents: what matters is the ACTION (understanding mapped code), not the SOURCE of the instruction (skill, plan, user, workflow). This prevents external workflows from overriding the graph protocol — agents must run `yg owner` + `yg build-context` before any action that involves understanding mapped code, regardless of what instructed it. The Failure States section reinforces this with source-agnostic language.

**Platform-specific installation:** Each agent platform has its own conventions for rules location. The platform.ts module centralizes this mapping so `yg init --platform <name>` works consistently across IDEs.

**Default config as template:** DEFAULT_CONFIG provides a minimal valid yg-config.yaml that adopters can customize. It mirrors yg-config.yaml schema but is a string constant for embedding. Defines 4 node types as an object keyed by type name (module, service, library, infrastructure), each with a description string explaining its purpose. No longer includes an artifacts section — the three standard artifacts (responsibility.md, interface.md, internals.md) are hardcoded as STANDARD_ARTIFACTS in cli/model. The `infrastructure` node type was added for guards, middleware, interceptors, and other components that affect blast radius but are invisible in call graphs. Node type descriptions come from config (not hardcoded in rules) so adopters can customize them.

**Simplified Quick Start with build-context --file:** Replaced the multi-step decision tree (step 0: maybe select, step 1: owner, step 2: choose between build-context/impact/both) with a single command: `yg build-context --file <path>`. Chose one-command flow over decision tree because agents simplify multi-step protocols to the first action and drop conditionals — fewer steps = fewer steps skipped. The `--file` flag resolves owner internally, eliminating the separate `yg owner` call. `yg select --task` remains available for when agents don't know which file to start from, but it's no longer step 0 of a mandatory sequence.

**Research-based structural improvements (v2.8.0 second pass):** Applied findings from LLM instruction-following research. (1) `<critical_protocol>` XML tags around the key rule — Claude trained to attend to XML structure (Anthropic Context Engineering Guide). (2) Verbatim repetition of critical rule at end of file — exploits recency effect; Google Research found prompt repetition improved 47/70 benchmarks with zero losses. (3) Few-shot example (correct vs wrong) — Anthropic: "examples are the pictures worth a thousand words." (4) Positive framing: NEVER list → ALWAYS list — avoids Ironic Process Theory / Pink Elephant Problem (16x Eval 2025, Anthropic docs). (5) Structured output requirement: agents must state `graph: <node>` or `graph: unmapped` before source code reasoning — converts self-audit from optional reflection to required output step (Nature npj Digital Medicine: self-audit works when explicitly triggered, not when left to agent initiative). (6) Authoritative + helpful framing: "required because it prevents cascading errors" instead of guilt-based "you're costing the user" — pragmatic framing research found authority + helpfulness alignment outperforms emotional leverage.

**Section reorder based on "Lost in the Middle" research (v2.8.0 third pass):** Restructured from CORE PROTOCOL → OPERATIONS → KNOWLEDGE BASE to PROTOCOL → REFERENCE → GUARD RAILS. The key change: procedures (Modify Source Code, Example, Conversation Lifecycle) moved from the middle dead zone to the primacy zone alongside the critical rule. Reference material (Graph Structure, Artifact Structure, CLI Reference) moved to the middle — safe in the dead zone because it's looked up, not internalized. Guard rails (Core Rules, Evasion Table, Failure States, edge-case procedures) moved to recency zone — agent has them fresh in memory when starting work. Rationale: "Lost in the Middle" (Liu et al., Stanford, TACL 2024) showed >30% attention drop in middle positions. Previously the Modify Source Code procedure was in this dead zone while meta-rules (evasion table) occupied the primacy zone. Chose to swap: procedures at edges (agent internalizes the steps), diagnostics at recency (agent has warnings fresh), reference in middle (lookup-only material tolerates lower attention).

**Preflight exception removed:** The original rules had "Exception: read-only requests (explain, analyze) — skip preflight." This was exploited as a loophole: agents classified code analysis tasks as "read-only" and skipped preflight, then read mapped source files without graph context. The exception is unfixable by rewording because agents cannot know if content is mapped without running yg — so any "skip if unmapped" exception requires the check it's trying to skip. Chose to remove the exception entirely over narrowing it.

**Subagent delegation rule:** CORE_PROTOCOL includes a "Delegating to Subagents" section requiring agents to prepend `.yggdrasil/agent-rules.md` as the first instruction when spawning subagents, plus explicit deliverables (working code + graph nodes + validate passing). Chose explicit instruction injection + acceptance criteria over relying on subagent auto-discovery because subagents do not inherit parent context and would otherwise read code without graph coverage, breaking graph-code consistency. Reading rules alone proved insufficient — subagents still deferred graph work without explicit deliverables. Alternative considered: embedding full rules in each subagent prompt — rejected because rules are long and the agent-rules.md file is already materialized on disk. The parent agent also gets a verification step: check for new/modified source files without graph coverage before accepting subagent output.

**Graph as specification (Rule 2 reframe):** Rule 2 was rewritten from "Code and graph are one" (a synchronization procedure) to "The graph is the specification; code implements it" (a philosophical principle). Chose the specification framing over the synchronization framing because agents treated the graph as documentation (written after code) rather than specification (written alongside or before code). The new rule establishes the graph as a "black hole" for knowledge — absorbing information from every source (external docs, conversations, decisions) and being self-sufficient. Timing tightened from "same response" to "before moving to the next file" to prevent batching. Planning rule added: graph updates are part of each step's definition of done, never a separate phase — this is tool-agnostic and applies to any planning workflow.

**Flow creation enforcement (v2.10.0):** Agents skipped flow creation during greenfield implementation — they built nodes and aspects but treated business processes as optional afterthoughts. Root cause: agents dismissed single-actor workflows (blog publishing, portfolio management) as "just CRUD" rather than goal-directed business processes. The existing flow identification guidance only mentioned multi-actor processes. Fix: expanded flow identification heuristic to explicitly cover single-actor workflows; added flow verification from specs (mandatory check); added flow participant maintenance rule (update flow nodes after splits); added step 4b to Modify Source Code checklist; added 3 evasion patterns and 3 failure states. Chose to broaden the heuristic test from "sequence of actions by different actors" to "goal-directed sequence of steps that a future agent needs to understand as a whole" — this captures both multi-actor and single-actor workflows.

**Task-level graph trigger (v2.12.0):** Added a task-level trigger alongside the existing file-level trigger. Root cause: when agents enter brainstorming/design/planning modes, the file-level trigger (`yg build-context --file`) doesn't fire because agents don't think of themselves as "interacting with a source file" — they think "I'm exploring ideas." Even when they read source files for research, they rationalize: "I'm just scoping." Real-world impact: an agent designed a 4-feature admin panel by reading 6+ source files without graph context, missing an admin-ux-rules aspect, a GalleryDelivery flow, and database conventions. Fix: six changes — (1) expanded `critical_protocol` to lead with `yg select --task "<goal>"` before any task, (2) added task-level entry to Quick Start, (3) added brainstorming correct/wrong example pair, (4) closed Context Assembly loophole by removing "the map alone is sufficient" escape hatch and adding measured cost (~8K tokens), (5) added 3 evasion patterns ("context too large", "I have a plan", "user told me what to do"), (6) added failure state for brainstorming without graph as first item in the list. Chose `yg select --task` as the task-level command because it's goal-oriented (fires before the agent knows which files to read) and already exists in the CLI. The task-level trigger is placed FIRST in critical_protocol for maximum visibility. The existing file-level trigger remains — both apply independently. The Context Assembly change was based on measured token counts from a real project (GalleryManager: 7750, HeroSection: 7847) — agents claimed packages were "too large" but they're smaller than a single source file.

**Spec ingestion & non-code knowledge (v2.9.0):** Added three new sections: Working from External Specifications, Non-Code Knowledge, Aspect Discovery During Implementation. Root cause: during full-system implementation from external specs, the agent captured only ~30% of spec knowledge — all technical, zero business context (personas, pricing, UX rationale, quality targets). The protocol was entirely file-centric (`build-context --file`, "after modifying source file") so knowledge without a source file had no capture trigger. Fix: explicit spec-to-graph phase before coding, conversation knowledge capture mandate, aspect discovery checkpoint after 3 features, expanded completeness test with business context check, 6 new information routing entries, 6 new evasion patterns, 4 new failure states, and a node sizing rule (max 10 files or 3 workflows per node). Chose checkpoint-at-3 over end-of-task extraction because after 30 features the rationale is forgotten and aspects become mechanical extractions without WHY.

**Knowledge preservation under budget pressure:** Error Recovery now explicitly handles W005 (budget warning) — agents must split nodes, never delete artifact content to reduce token count. Added to Failure States as an anti-pattern. The validator now separates informational from actionable warnings: W005/W006 show the full per-category breakdown (own, hierarchy, aspects, flows, dependencies) so agents can diagnose where budget is consumed; W015 (own-budget-warning) is the actionable warning that fires when own-layer tokens exceed 50% of the warning threshold, directing agents to split nodes. Rules Error Recovery section updated to reflect this: W005/W006 are informational with breakdown, W015 is actionable for own content. Chose explicit prohibition + procedural guidance (identify subset, propose split, create children, redistribute) over a simple "don't delete" because agents need a concrete alternative action.
